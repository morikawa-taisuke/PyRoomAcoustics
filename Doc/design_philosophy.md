# PyRoomAcoustics リポジトリ設計思想

本ドキュメントでは、本リポジトリのディレクトリ構成・モジュール分割の「設計思想（なぜこのように作られているか）」を後輩や共同開発者向けに解説します。

---

## 1. 全体的な設計方針（責務の分離）

本リポジトリは**「実行用スクリプト」と「裏側の共通処理（コアライブラリ）」を完全に分離する**という設計思想に基づいています。
これにより、「何でも入っている巨大なユーティリティ（Fat Utility）」の発生を防ぎ、どこに何を書くべきかがひと目で分かる状態を維持します。

## 2. ディレクトリ構成と役割

リポジトリは大きく分けて以下の3つの要素で構成されています。

### 📁 `config/` (設定情報の管理)
*   **役割**: コード自身を変更せずに、実行時の振る舞いを切り替えるための場所です。
*   **思想**: 部屋の寸法や目標のSNR、入出力ファイルの構成などはすべて `dataset_config.yaml` などのYAMLファイルで一元管理されています。Pythonコード内に直接マジックナンバー（固定値やパス）を書かないことを徹底しています。

### 📁 `scripts/` (実行用エントリーポイント)
*   **役割**: ユーザーが直接ターミナルから `uv run python scripts/...` のように実行する「メインスクリプト」置き場です。
*   **思想**: `generate_dataset.py` や `resample_dataset.py` など、処理を開始する発火点となるファイルのみを置きます。ここには共通関数（他のファイルから `import` されるような使い回す関数）は**定義しません**。

### 📁 `src/core/` (共通コアライブラリ)
*   **役割**: `scripts` から呼び出される裏側のシステム・処理ロジックです。
*   **思想**: 複数のスクリプトから再利用される関数群を格納します。処理の性質（ジャンル）ごとに細かくファイルが分割されています。

---

## 3. `src/core/` 内のモジュール分割ルール

コアライブラリは、機能の重複を防ぐために「どんな処理を行うか」によって厳密にファイルが分けられています。
新しく関数を作る際は、**以下のどれに該当するかを考えて適切なファイルに追記してください。**

### 📄 `audio.py` 【純粋な音声波形処理】
*   **対象**: 音性の読み込み(`sf.read`)、書き込み(`sf.write`)、振幅の正規化、クロップ処理、SNRの計算と重畳など。
*   **特徴**: 空間シミュレーション（PyRoomAcoustics）には一切依存せず、1次元または2次元のNumpy配列を扱う純粋な「波形データ操作」だけを担当します。

### 📄 `rec_utility.py` 【空間シミュレーション処理】
*   **対象**: 部屋(ShoeBox)の構築、マイクアレイの座標計算、音源の配置、RIR（インパルス応答）の計算と畳み込み処理など。
*   **特徴**: PyRoomAcoustics (`pa`) を使った複雑な音響シミュレーションに特化した関数群です。「音声波形そのもの」の操作は `audio.py` に任せ、こちらは「空間と残響の計算」に専念します。

### 📄 `const.py` 【グローバル定数・絶対パス】
*   **対象**: OSやPC環境に依存するような絶対パスのルート（`SOUND_DATA_DIR`など）や、システム全体で共有するパス情報を定義します。
*   **特徴**: 実行環境やPCが変わった際に、このファイルさえ直せば全てのスクリプトが動くようにするための起点となる最重要ファイルです。

### 📄 `reverb_feature.py` 【音響特徴量計算】
*   **対象**: RIR（インパルス応答）から `C50` や `D50` などの物理的・音響学的な特徴量を計算する解析処理を担当します。

### 📄 `rec_config.py` 【シミュレーション用の定数パラメータ】
*   **対象**: サンプリングレート（`16000`）など、リポジトリ全体を通じて「不変であるべき」音響システムの設定値を保持します。

---

## 4. この設計を守ることのメリット

1. **バグ発生率の低下**: 例えば「音圧の計算式」を直したい場合、以前のように複数ファイルに同じ関数がコピーされていると修正漏れが発生します。関数を1箇所（`audio.py`）に集約することで、1箇所直せばプロジェクト全体が直るようになります。
2. **学習・引き継ぎコストの大幅な削減**: 後輩がプロジェクトに参加したとき、「実行したいなら `scripts/` を見る」「波形処理の便利な関数を使いたいなら `audio.py` を見る」という直感的な推測が成り立ち、コードを読み解く時間が大幅に減ります。
3. **安全な再利用**: 処理が完全に分離しているため、将来別の全く新しいシミュレーションスクリプトを書く場合でも、`from core.audio import save_wav` のように自信を持って再利用可能です。
