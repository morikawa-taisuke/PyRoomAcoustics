# 設定ファイルの管理ガイドライン（3層構造の設計思想）

本リポジトリでは、「どの設定ファイルを書き換えればいいのか分からない」「誤って別の数値を壊してしまった」といった事故を防ぐため、設定項目を**「変更される理由・頻度」に応じて以下の3つの階層に完全に分離**して管理しています。

新しい環境（別のPC）で実行する際や、新たな実験パラメータを追加する際は、必ず以下のルールに従って適切な階層のファイルを編集してください。

---

## 🏗️ 第1層：ローカル環境設定（`.env`）
**「PCや開発者ごとに異なる、自分専用の絶対パス」**を定義する最上位の層です。

*   **役割**: Dドライブやユーザーディレクトリなど、環境によって必ず変わる「データの大元となる根っこのパス」を設定します。
*   **ファイル場所**: プロジェクト直下（`PyRoomAcoustics/.env`）※各人が自分で作成します。
*   **中身の例**: `SOUND_DATA_DIR=D:/sound_data/`
*   **ルール**:
    *   このファイルは後輩がリポジトリをダウンロード（clone）した直後に、必ず自分自身の環境に合わせて一番最初に作成します。
    *   **絶対にGit（リモートリポジトリ）にはプッシュしません**（`.gitignore` に登録済み）。個人のPC環境が他人に漏れたり、チーム全員の設定を破壊するのを防ぐためです。

---

## 🛠️ 第2層：実験パラメータ（`config/*.yaml`）
**「どんなデータセットを作るか、部屋の広さや残響はどうするか」**といった、スクリプト実行時に切り替えるパラメータを定義する層です。

*   **役割**: 実験の条件（入力データ、SNR、残響時間、マイク位置など）を決定します。
*   **ファイル場所**: `config/dataset_config.yaml` などのYAMLファイル。
*   **中身の例**: 部屋の寸法 `[5.0, 5.0, 5.0]`、SNR `[0.0, 10.0]`、マルチプロセス数 `6` など。
*   **ルール**:
    *   設定項目の中には「絶対パス」を直書きしてはいけません。第1層のルートパスからの「相対パス」または「ディレクトリ名（例: `VCTK`）」のみを記述します。
    *   実行する際は、スクリプトの中でファイルをハードコードせず、必ずターミナルから引数として設定ファイルを渡します。
        （例: `uv run python scripts/generate_dataset.py --config config/dataset_config.yaml`）

---

## ⚙️ 第3層：システム定数と内部構造（`src/core/` 内のPythonファイル）
**「システムが破綻しないための絶対不変のルール」**を定義する最下層です。ここには、後輩が日常の運用で**絶対に書き換えてはいけない**定数群が配置されています。

### ① `const.py`（システム内部構造の固定化）
*   **役割**: 第1層（`.env`）で読み込んだ「根っこのパス」を元に、`/sample_data` や `/speaker` といった変更してはならない**システム内部の相対構造**をガッチリと構築します。
    （例: `SAMPLE_DATA_DIR = SOUND_DATA_DIR / 'sample_data'`）
*   **理由**: 誰のPCであってもディレクトリの内部構造のルール（仕様）は必ず同じになるため、この構造自体に後輩が介入してバグを生むリスクをゼロにします。

### ② `rec_config.py` または `dsp_params.py`（物理・数学ルール）
*   **役割**: サンプリングレート(`16000Hz`)や、高速フーリエ変換のフレームサイズ(`1024`)など、システムが前提としている**物理学的な不変の定数**を定義します。
*   **理由**: これらの値は音響シミュレーションの心髄であり、ここが少しでもズレると出力データが使い物にならなくなります。そのため、環境構築の定数（第1層）や実験パラメータ（第2層）とは完全に隔離された聖域として扱います。

---

## 🌟 なぜこのように管理するのか（最大のメリット）

1.  **環境構築が1分で終わる**:
    新しいPCに移行したときや、新しい後輩が参加したとき、**「`.env` に1行だけ自分のパスを書く」** だけで、システム全体の全てのスクリプトが即座に起動するようになります。他のPythonファイルの中を探し回る必要は一切ありません。
2.  **事故の完全防止**:
    「パスを直そうとして、うっかりサンプリングレートを16000から書き換えてしまった」というような人為的ミス（ヒューマンエラー）が物理的に起こらなくなります。
3.  **DRY（Don't Repeat Yourself）原則の徹底**:
    パスの起点が `.env` という単一の情報源（Single Source of Truth）に集約されているため、DドライブからCドライブへデータを移動させた際も、たった1箇所の修正でプロジェクト全体が追従します。
